\chapter{Methodology}

\section{Development Approach}
\subsection*{Agile Workflow}
The development of \textit{TRiM} followed an agile methodology, incorporating several techniques to ensure continuous progress and stakeholder alignment. A Kanban board was maintained using GitHub Projects to track all development work. Each item represented a unit of work from a user story and moved through \textit{Backlog}, \textit{In Progress}, and \textit{Done} columns as development progressed, see Figure~\ref{fig:kanban-board}.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.4\textwidth]{images/kanbanboard.png}
    \caption{GitHub Projects Kanban board used to track development progress.}
    \label{fig:kanban-board}
\end{figure}

Weekly meetings were held with the project supervisor to review progress, discuss technical decisions, and receive guidance. In parallel, regular meetings were carried out with the client, Jean, the owner of V7 Barbers. All meetings were documented for reference.

\subsection*{Version Control and Development Tools}
All source code was managed using Git and hosted on GitHub. Commits were made frequently to maintain a clear history of changes throughout the project. Development was carried out primarily using IntelliJ IDEA for the Spring Boot backend and Visual Studio Code for the React frontend. GitHub Codespaces was also used on occasion for remote development when working from a different machine. Docker Compose was used locally to run the PostgreSQL database alongside the application during development.

\subsection*{Requirements Gathering}
Working with a real client was key to this project. Jean, the owner of V7 Barbers, participated in regular meetings that served as the primary mechanism for gathering and validating requirements. The first few sessions focused on understanding the day-to-day challenges of running a walk-in-only barber shop: limited visibility into peak times, no way to manage no-show customers, and the burden of manual bookkeeping. These conversations shaped the initial user stories and ensured the system addressed genuine business problems.

As development progressed, client meetings shifted more towardrs demonstration. Live walkthroughs of the system prompted feedback that hadn't been thought of before like deposit-based payments, where customers pay a percentage upfront and settle the balance in-shop, and customer blacklisting to protect the business from repeated no-shows.

Requirements were captured as user stories, each documented as a GitHub Issue following the same structure. Every user story included four sections: a \textbf{User Story} statement written in the standard ``As a [role], I want [feature], so that [benefit]'' format; a set of \textbf{Acceptance Criteria} expressed using Given/When/Then conditions; a \textbf{Priority} classification using the MoSCoW method (Must Have, Should Have, Could Have); and \textbf{Technical Notes} outlining relevant implementation details such as API endpoints and UI behaviour. Figure~\ref{fig:user-story-example} illustrates a user story from the project.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.7\textwidth]{images/user-story-example.png}
    \caption{Example user story with acceptance criteria, priority, and technical notes.}
    \label{fig:user-story-example}

Priorities were assigned based on a combination of client input, technical dependencies, and time constraints. Must Have items represented core functionality without which the system would not work, while Should Have and Could Have items were developed after the essential features were in place. Each user story mapped directly to a card on the Kanban board, providing traceability for each requirement.

\section{Generative AI Usage}
\subsection*{Code Generation in VSCode}
GitHub Copilot was used as an inline code auto-complete in Visual Studio Code. Its primary value was in speeding up the production of boilerplate code that makes up a significant portion of any full-stack Spring Boot and React application.

On the backend, Copilot was effective at suggesting Java entity classes with JPA annotations, DTOs with getter and setter methods, repository interfaces, and repetitive controller patterns such as dependency injection, service delegation, and \texttt{ResponseEntity} wrapping. On the frontend, it assisted with TypeScript interfaces that mirrored backend DTOs, Axios-based API service methods, and form validation logic.

This was most useful during the early stages of development when the foundational structure of entities, repositories, controllers, and base components was being established. Copilot's suggestions helped speed up this process, allowing more focus to shift toward the business logic that differentiated the system, such as the availability algorithm, deposit payment flow, and customer blacklisting. All suggestions were reviewed and adjusted before being accepted.

\subsection*{Testing Automation with Claude Code}
Claude Code CLI was used to generate the integration test suite for the backend API. The test files were generated one at a time, with each file targeting a specific controller or operation. This allowed each test class to be reviewed individually to ensure it covered the correct functionality and edge cases.

The resulting suite consists of 15 integration test classes covering areas such as authentication, booking creation and cancellation, barber availability, payment webhooks, and dashboard endpoints. Each test class runs against an in-memory H2 database, sets up its own test data, and makes real HTTP requests through Spring's \texttt{MockMvc}. Claude Code's ability to generate the tests quickly allowed for both a much broader test suite and also quick iterations to either refine the tests or fix bugs that were found during testing.

\subsection*{Code Review}
Claude Code was also used to review existing code for security vulnerabilities and performance issues. This involved pointing the tool at specific files or modules and having it analyse the implementation for common weaknesses such as unvalidated input, improper access control, inefficient database queries, unnecessary object creation, or even typos. Findings were not applied automatically; instead, each suggestion was worked through iteratively, discussing the reasoning behind the change before deciding whether to apply it. This back-and-forth process helped ensure the final code was both clean and well-understood.

\section{Research Approach}
\subsection*{Literature Search}
The literature review was done across IEEE Xplore, ACM Digital Library, and ATU's library database, covering research published between 2008 and 2025. A combination of broad and refined Boolean search queries was used, such as \texttt{multitenant database architecture}, \texttt{multi-tenant SaaS database design}, and \texttt{multi-tenan* AND (architecture OR design OR patterns)}.

Broad queries returned between 1,100 and 2,500 results, while more refined queries narrowed this to approximately 900. Duplicates across databases were removed, and the remaining papers were screened by reviewing the title, abstract, introduction, and conclusion. Papers that did not directly address database design patterns were discarded at this stage. This process resulted in a selection of 13 peer-reviewed papers.

Papers were included if they discussed database design patterns within multi-tenant architecture and addressed areas such as security, scalability, performance, or cost. Papers that were not peer-reviewed, not available in English, or that tackled multi-tenancy without focusing on database design were excluded.

\subsection*{Choosing the Technology Stack}
The backend was built with Java and Spring Boot, chosen for its wide adoption in SaaS development and its robust security, data access, and REST APIs. Spring Security and JWT handled authentication, Spring Data JPA managed database interaction, and Spring Validation was used for input handling. PostgreSQL was selected as the database due to its support for Row-Level Security, which was central to the multi-tenant data isolation strategy explored in the literature review and applied in this project.

The frontend was built with React, TypeScript, and Vite. React was chosen for its component-based architecture, which suited the application's multiple user-facing views across customer, barber, and admin roles. TypeScript was used to catch type errors early and maintain consistency with the backend's strongly typed DTOs. Supporting libraries such as Redux Toolkit, React Router, Axios, and Tailwind CSS were selected based on compatibility with the React ecosystem. Stripe's React SDK was used for the payment integration.

Docker Compose was used to manage three containers: a PostgreSQL 16 database, the Spring Boot API, and the React frontend. This setup ensured a consistent environment between local development and the production VPS, and simplified deployment by defining the entire stack in a single configuration file.

\section{Evaluation Methodology}
\subsection*{Benchmarking With Gatling Load Testing}
Gatling was used to load test the API under simulated real-world conditions. The simulation defines six scenarios that mirror the application's core user flows: public browsing, customer registration, the full booking lifecycle, admin management, barber operations, and an availability stress test. Each scenario combines read and write operations to reflect realistic usage rather than testing endpoints in isolation.

The load is distributed across the scenarios to approximate expected traffic patterns, with public browsing and the booking flow receiving the largest share. Users are ramped up over 60 seconds and then held at a constant level for 90 seconds. The total number of concurrent users is configurable, defaulting to 100. Assertions are set on the 95th and 99th percentile response times and the overall success rate. The simulation was run twice under the same conditions: once with Row-Level Security enabled and once without, in order to measure the performance overhead introduced by RLS. The simulation relies on a pre-seeded database, described in the following subsection. Figure~\ref{image:apiBenchmarking} shows the simulation being executed from IntelliJ.

\subsection*{Test Data Generation With DataSeeder}
To run the Gatling simulation against a realistic dataset, a custom \texttt{DataSeeder} component was built using Spring Boot and the DataFaker library. The seeder is activated via a Spring profile and populates the database with configurable volumes of businesses, customers, barbers, services, and bookings using JDBC batch inserts for performance. Default values generate 5 businesses, 50,000 customers and 100,000 bookings per business, along with barbers, service categories, and realistic scheduling data such as availability windows, breaks, and booking time slots.

This seeded data served two purposes: it provided the volume needed for meaningful load testing with Gatling, and it ensured that the simulation's feeders could reference valid entities such as barber IDs, service IDs, and customer credentials.

\subsection*{Performance Metrics}
The following metrics were used to evaluate API performance across both test runs (with and without RLS): response time at the 95th and 99th percentiles, overall success rate (percentage of requests returning expected status codes), and throughput measured in requests per second. These metrics are reported automatically by Gatling at the end of each simulation run, broken down per request, as shown in Figure~\ref{fig:gatling-report}. The results are explored in detail in a later chapter and were compared between the two configurations to quantify the overhead introduced by Row-Level Security.
