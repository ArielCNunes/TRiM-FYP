\section{Development Approach}
\subsection*{Agile Workflow}
The development of \textit{TRiM} followed an agile methodology, incorporating several techniques to ensure continuous progress and stakeholder alignment. A Kanban board was maintained using GitHub Projects to track all development work. Each item represented a unit of work from a user story and moved through \textit{Backlog}, \textit{In Progress}, and \textit{Done} columns as development progressed, see Figure~\ref{fig:kanban-board}.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.7\textwidth]{images/kanbanboard.png}
    \caption{GitHub Projects Kanban board used to track development progress.}
    \label{fig:kanban-board}
\end{figure}

Weekly meetings were held with the project supervisor to review progress, discuss technical decisions, and receive guidance. In parallel, regular meetings were conducted with the client, Jean, the owner of V7 Barbers. All meetings were documented for reference.

\subsection*{Version Control and Development Tools}
All source code was managed using Git and hosted on GitHub. Commits were made frequently to maintain a clear history of changes throughout the project. Development was carried out primarily using IntelliJ IDEA for the Spring Boot backend and Visual Studio Code for the React frontend. GitHub Codespaces was also used on occasion for remote development when working from a different machine. Docker Compose was used locally to run the PostgreSQL database alongside the application during development.

\subsection*{Requirements Gathering}
Working with a real client was central to this project. Jean, the owner of V7 Barbers, participated in regular meetings that served as the primary mechanism for gathering and validating requirements. Early sessions focused on understanding the day-to-day challenges of running a walk-in-only barber shop: limited visibility into peak times, no way to manage no-show customers, and the burden of manual bookkeeping. These conversations shaped the initial user stories and ensured the system addressed genuine business problems.

As development progressed, client meetings shifted more towardrs demonstration. Live walkthroughs of the system prompted feedback that hadn't been thought of before like deposit-based payments, where customers pay a percentage upfront and settle the balance in-shop, and customer blacklisting to protect the business from repeated no-shows.

Requirements were captured as user stories, each documented as a GitHub Issue following a consistent structure. Every user story included four sections: a \textbf{User Story} statement written in the standard ``As a [role], I want [feature], so that [benefit]'' format; a set of \textbf{Acceptance Criteria} expressed using Given/When/Then conditions; a \textbf{Priority} classification using the MoSCoW method (Must Have, Should Have, Could Have); and \textbf{Technical Notes} outlining relevant implementation details such as API endpoints and UI behaviour. Figure~\ref{fig:user-story-example} illustrates a user story from the project.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.7\textwidth]{images/user-story-example.png}
    \caption{Example user story with acceptance criteria, priority, and technical notes.}
    \label{fig:user-story-example}
\end{figure}

Priorities were assigned based on a combination of client input, technical dependencies, and time constraints. Must Have items represented core functionality without which the system would not work, while Should Have and Could Have items were developed after the essential features were in place. Each user story mapped directly to a card on the Kanban board, providing traceability from requirement through to implementation and acceptance.
